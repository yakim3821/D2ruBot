# DotaForumBot Roadmap

## Цель первой версии

Бот работает в разделе `Таверна`: `https://dota2.ru/forum/forums/taverna.6/`.

Поведение первой версии:
- бот отслеживает новые темы в разделе;
- бот отвечает только в новые темы;
- бот публикует в теме только один раз;
- после первого ответа бот в эту тему больше не возвращается.

До подключения LLM делаем только механику форума, хранение данных и бизнес-логику.

## Зафиксированные решения

- Рабочий раздел: `Таверна`
- URL раздела: `https://dota2.ru/forum/forums/taverna.6/`
- Правило ответа: один ответ на новую тему
- Повторные ответы в ту же тему запрещены
- Базовый стиль: нейтральный
- Основа будущей личности: посты пользователя `yakim38`
- Источник постов для будущего профиля: `https://dota2.ru/forum/members/yakim38.815329/activity/posts/`
- Основное хранилище: `PostgreSQL`
- LLM на первом этапе не подключается
- Безопасная тестовая площадка: личная переписка `https://dota2.ru/forum/conversation/123.1039856/`

## Этапы

### 1. Транспортный слой форума

Нужно обеспечить стабильную работу низкоуровневых операций:
- авторизация;
- восстановление сессии из `session.json`;
- повторный логин при истечении сессии;
- чтение страниц разделов;
- чтение страниц тем;
- чтение страниц переписок;
- отправка сообщений в тему;
- отправка сообщений в переписку.

### 2. Интеграция с PostgreSQL

Нужно подключить PostgreSQL и подготовить базовую схему данных.

Планируемые сущности:
- `topics`
- `posts`
- `users`
- `bot_replies`
- `user_style_profiles`
- `scan_state`

### 3. Парсер раздела `Таверна`

Бот должен уметь:
- читать список тем в разделе;
- извлекать `topic_id`;
- извлекать URL темы;
- извлекать заголовок темы;
- извлекать автора темы;
- извлекать дату создания или доступный временной маркер;
- определять, новая тема или уже известная.

### 4. Парсер темы

Бот должен уметь:
- открывать тему;
- читать стартовый пост;
- извлекать автора темы;
- извлекать текст стартового поста;
- извлекать метаданные темы;
- сохранять тему и стартовый пост в БД.

### 5. Бизнес-правило "один ответ на тему"

Нужно реализовать жесткое правило:
- если тема новая и бот еще не отвечал, тема может быть обработана;
- если бот уже отвечал, тема больше никогда не обрабатывается повторно;
- признак ответа должен фиксироваться в БД;
- повторная публикация должна быть невозможна даже после перезапуска процесса.

### 6. Базовая персона без LLM

На этом этапе бот не имитирует пользователя через модель.

Нужно подготовить фундамент:
- собрать посты пользователя `yakim38`;
- сохранить их в БД;
- позже на их основе построить summary-профиль;
- пока использовать нейтральный стиль и простые шаблоны или правила генерации ответа.

### 7. Контур безопасного тестирования

До публикации в реальные темы все тесты проводятся в личной переписке:
- тексты;
- логика ответа;
- проверка транспортного слоя;
- проверка антидублей;
- проверка шаблонов.

Тестовый URL:
- `https://dota2.ru/forum/conversation/123.1039856/`

### 8. Подготовка к подключению LLM

После завершения инфраструктурного этапа:
- формируем профиль личности на основе постов `yakim38`;
- описываем стиль в структурированном виде;
- подключаем LLM только после того, как форумная логика и БД полностью готовы;
- используем LLM только для генерации первого ответа в новую тему.

## Рабочий порядок разработки

### Спринт 1

- добить стабильную отправку сообщений в темы и переписки;
- добить чтение страниц форума;
- стабилизировать сессии и `session.json`.

### Спринт 2

- подключить PostgreSQL;
- создать схему таблиц;
- сохранить служебное состояние бота.

### Спринт 3

- реализовать парсер раздела `Таверна`;
- сохранять найденные темы в БД;
- определять новые темы.

### Спринт 4

- реализовать парсер конкретной темы;
- сохранять стартовый пост;
- маркировать темы, в которых бот уже отвечал.

### Спринт 5

- реализовать логику одного ответа на тему;
- сначала прогонять ответы через личную переписку;
- только после проверки включать боевой режим.

### Спринт 6

- собрать посты `yakim38`;
- подготовить данные для будущего профиля личности;
- перейти к этапу генерации ответов.

## Что нужно от пользователя

Для следующих этапов потребуется:
- доступ к PostgreSQL:
  - host
  - port
  - database
  - user
  - password
- подтверждение режима запуска:
  - polling-скрипт
  - или постоянный worker
- подтверждение интервала проверки раздела

Рекомендуемый стартовый режим:
- polling раз в 2 минуты.

## Ограничения первой версии

Первая версия не должна:
- отвечать в старые темы;
- отвечать в одну и ту же тему дважды;
- работать по всему форуму;
- использовать LLM до готовности транспортного слоя и БД;
- имитировать пользователей напрямую без промежуточного профиля.
